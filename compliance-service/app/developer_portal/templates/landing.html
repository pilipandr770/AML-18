Ein Baustein, drei regulatorische Anforderungen

AML+18 ist das rechtliche Fundament, auf dem Ihr Krypto-Projekt aufbauen kann — nicht ein fertiges Produkt für ein bestimmtes Geschäftsmodell. Es macht aus einer bloßen Blockchain-Adresse eine verifizierte Entität: Sanktionsprüfung, Nachweis der Wallet-Kontrolle, Altersverifikation. Was Sie danach damit bauen — Börse, Wallet, Umtauschdienst, Plattform — ist Ihre Sache; Lizenzen und Genehmigungen für Ihr konkretes Geschäftsmodell liegen außerhalb dieses Bausteins.

Travel-Rule-Screening

Sanktionsprüfung bei Krypto-Transfers zwischen zwei VASPs, über das echte TRISA-Protokoll. Erfordert einen eigenen TRISA-Knoten — keine REST-Registrierung hier.

Wallet-Ownership-Verification

Nachweis der Kontrolle über eine Self-Hosted-Wallet-Adresse bei Transfers ab einem Schwellenwert. REST-API, sofort einsatzbereit.

Altersverifikation

Privatsphärenschonender Alters-Ja/Nein-Nachweis, für Adult-Content genauso wie für Online-Glücksspiel. REST-API, sofort einsatzbereit.

REST-Einstieg (Wallet-Ownership & Altersverifikation)

Nach der Registrierung erhalten Sie einen API-Schlüssel. Jeder Aufruf trägt ihn als Bearer-Token:

Wallet-Ownership-Verification

# Muss dieser Transfer überhaupt geprüft werden?
curl "http://localhost:8300/wallet-ownership/requirement?transfer_amount_eur=1500" \
  -H "Authorization: Bearer aml18_sk_..."

# Challenge anfordern, vom Kunden mit dem Wallet-Key signieren lassen
curl -X POST http://localhost:8300/wallet-ownership/challenges \
  -H "Authorization: Bearer aml18_sk_..." \
  -H "Content-Type: application/json" \
  -d '{"network": "ETH", "address": "0x..."}'

# Signatur einreichen
curl -X POST http://localhost:8300/wallet-ownership/verifications \
  -H "Authorization: Bearer aml18_sk_..." \
  -H "Content-Type: application/json" \
  -d '{"method": "signed_message", "challenge_id": "...", "signature": "0x..."}'

Altersverifikation

curl -X POST http://localhost:8300/age-verify/check \
  -H "Authorization: Bearer aml18_sk_..." \
  -H "Content-Type: application/json" \
  -d '{"subject_reference": "platform-user-123", "proof_token": "mock:over18"}'

Vollständige Endpunkt-Referenz: siehe README.md des Projekts.

Travel-Rule-Einstieg (TRISA)

Für echtes Sanktions-Screening bei VASP-zu-VASP-Transfers betreiben Sie selbst einen TRISA-Knoten und verbinden ihn über das offizielle Protokoll — kein API-Schlüssel dieser Seite ist dafür nötig oder ausreichend. Details und ein funktionierendes Demo-Skript: examples/travel-rule-demo/README.md im Projekt-Repository.
